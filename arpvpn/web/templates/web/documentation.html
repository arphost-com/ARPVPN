TLS and access
What ARPVPN covers

ARPVPN manages WireGuard interfaces, client peers, site-to-site peers, TLS/web access, user roles, traffic collection, RRD graphs, statistics, and selected automation APIs from one interface.

Clients should use this platform mainly for connection details and bandwidth visibility. Staff users can also manage VPN lifecycle, users, certificates, and troubleshooting workflows.

Getting Started
  1. Complete the initial setup page and verify the public endpoint.
  2. Create at least one WireGuard interface with a reachable UDP port.
  3. Add client peers or site-to-site peers under the interface.
  4. Use the statistics page to confirm handshakes, transfer, and graph history.
  5. Enable TLS after HTTP access is working and choose whether HTTP should redirect.
Roles and Access
  • Admin: full local server control, users, TLS, WireGuard, and API access.
  • Support: user/client operations and client impersonation, but limited role changes.
  • Client: view only their connection details, usage, downloads, and scoped statistics.
Site-to-site Quick Guide
  1. Create or choose a WireGuard interface in WireGuard.
  2. Add a peer and set Mode to Site-to-site.
  3. Set the tunnel IP for the remote gateway peer.
  4. Add remote LANs (CIDR) in Remote site subnets.
  5. Download the generated config and apply it on the remote gateway.
  6. Ensure both firewalls allow UDP on the interface listen port.
ARPVPN will automatically include the remote site subnets in server-side `AllowedIPs`.

Client VPN workflow
  1. Open WireGuard and create an interface if one does not exist.
  2. Add a peer in Client mode.
  3. Download the generated peer config and import it into the client device.
  4. Use Statistics to verify handshake age, traffic, and alerts.

Web Access and TLS
  • HTTP: plain web access, typically for first-time setup or local environments.
  • Self-signed: ARPVPN generates and serves its own certificate.
  • Let's Encrypt: ARPVPN issues a public certificate for a real hostname.
  • Reverse proxy: ARPVPN stays behind another web server or load balancer.

If redirect is enabled, use HTTPS consistently. If redirect is disabled, both HTTP and HTTPS can remain available on the configured ports.

Global TLS changes are super-admin only. Tenant admins can store tenant-scoped TLS intent through the API for separate tenant runtime stacks.

Separate tenant runtime planning is exposed through /api/v1/tenants/<tenant_id>/runtime for port allocation and desired-state tracking.

Statistics and RRD Graphs
  • Use Statistics for connection history, rollups, alerts, failures, and RRD graph links.
  • The dashboard shows current session visibility and high-level health.
  • Each connection can expose historical graph images and JSON history endpoints when traffic collection is enabled.
  • Clients only see scoped data for their own connection; staff sees broader operational data.

API and Automation

ARPVPN exposes a versioned API under /api/v1. The current API is strongest for auth, tenant/user lifecycle, WireGuard control, statistics, RRD access, themes, and TLS management.

  • Cookie-authenticated API writes require a CSRF token from GET /api/v1/auth/csrf.
  • Bearer-token clients use POST /api/v1/auth/token and do not need API CSRF headers.
  • Account MFA is configured once from Profile and used for web login; ARPVPN does not maintain a separate WireGuard-specific 2FA stack.
  • Super admins can export or restore snapshots with /api/v1/system/backup and /api/v1/system/restore.
  • Setup, profile, about, network inventory, and restart flows also have API parity under /api/v1/setup/*, /api/v1/profile, /api/v1/about, /api/v1/network/inventory, and /api/v1/system/restart.
  • API groups can be feature-flagged with environment variables such as ARPVPN_FEATURE_API_AUTH=0 or ARPVPN_FEATURE_API_SYSTEM=0.
  • The repository now ships a generated Python client in sdk/python from the live OpenAPI contract.

Host-level route installs, ACL enforcement, and some rollout/operations pieces are still in progress.

Troubleshooting
  • If a peer shows stale or down, confirm endpoint reachability, UDP port exposure, and recent handshake time.
  • If RRD graphs are empty, verify traffic collection is enabled and the tunnel has transferred data over time.
  • If HTTP and HTTPS are both enabled, keep the same origin during login and setup workflows.
  • If site-to-site traffic fails, check forwarding, firewall rules, and that the remote LANs are included in the peer route configuration.

Client View

As a client, your main pages are the dashboard, statistics, and your downloadable connection information. Use those pages to confirm the tunnel is active and to review historical usage. Your account MFA is configured once in Profile and applies to web login; WireGuard tunnels themselves do not have a separate login prompt.

If you need user, route, or certificate changes, contact your tenant or platform administrator.
